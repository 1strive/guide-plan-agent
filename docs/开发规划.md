# 学习路线驱动的项目开发规划

> 基于你「有一定基础、快速迭代」的选择，本计划从阶段2（Prompt Engineering）开始重点投入，阶段1的核心概念通过代码实践快速覆盖。全周期约 **10 周**。

---

## 前置：阶段1 基础补强（第1周上半周，3~4天）

虽然你已有基础，但项目还需要补齐一些阶段1的**工程实践**，作为后续阶段的底座。

### Task 1.1：流式输出（Streaming）支持
- **学习目标**：理解推理参数（temperature、max_tokens）对输出的影响；掌握 SSE/流式 API 的设计。
- **项目实践**：在 `src/index.ts` 的 `/sessions/:id/chat` 接口中，增加 `POST /sessions/:id/chat/stream` 端点，使用 `text/event-stream` 返回 LLM 流式输出。
- **关联文件**：`src/index.ts`、`src/agent/llm.ts`
- **输出物**：可运行的流式对话接口；对比实验记录（不同 temperature 的输出差异笔记）。

### Task 1.2：Token 计数与成本控制
- **学习目标**：理解 Token、Tokenizer 概念；建立「按 token 计费」的工程意识。
- **项目实践**：在 `src/agent/llm.ts` 中添加请求/响应 token 估算逻辑（可先用字符数近似，再接入 tiktoken 或 OpenAI 返回的 usage 字段）；记录每次调用的 token 用量到日志。
- **关联文件**：`src/agent/llm.ts`、`src/config.ts`
- **输出物**：带用量统计的日志；单条请求成本计算笔记。

## 阶段2：Prompt Engineering（第1周下半周 ~ 第2周，约10天）

### Task 2.1：提示词版本化与结构化重构
- **学习目标**：掌握「系统提示 vs 用户消息」「角色-任务-约束-输出格式」的结构化写法；建立版本管理意识。
- **项目实践**：重构 `src/agent/prompts.ts`，从单字符串改为**提示词模板引擎**（支持变量插值），设计多版本提示词存储结构（如 `prompts/v1_base.ts`、`prompts/v2_cot.ts`）。
- **关联文件**：`src/agent/prompts.ts`
- **输出物**：结构化提示词模板；版本切换机制。

### Task 2.2：Few-shot 与 Chain-of-Thought（CoT）提示词
- **学习目标**：理解 Few-shot 示例选取原则；掌握 zero-shot CoT（「让我们一步步想」）的适用边界。
- **项目实践**：
  - 在提示词模板中增加 `examples` 字段，为旅游场景（如「推荐亲子游目的地」）注入 2~3 组 Few-shot 示例。
  - 增加 CoT 模式开关：在系统提示词末尾条件性附加「请逐步分析用户需求，再给出建议」。
- **关联文件**：`src/agent/prompts.ts`、`src/agent/llm.ts`
- **输出物**：两种提示词变体（标准版 vs CoT 版）。

### Task 2.3：Prompt 评测与迭代框架
- **学习目标**：建立「评测意识」——同一提示词用多组输入测试，记录失败案例并迭代。
- **项目实践**：
  - 在项目中新建 `src/eval/` 目录，创建测试集文件（10~30 条旅游问答用例）。
  - 写一个脚本 `scripts/eval-prompt.ts`，批量跑不同版本的提示词，输出准确率、失败案例列表。
  - 用 JSON 记录每次实验：版本号、参数、测试集、通过率、典型失败 case。
- **关联文件**：新增 `src/eval/testset.ts`、`scripts/eval-prompt.ts`
- **输出物**：可复用的 Prompt 评测流水线；至少完成 2 轮迭代记录。

## 阶段3：RAG 技术（第3周 ~ 第4周，约2周）

> 当前 `src/rag/` 为空，且系统提示词中提到了 `semantic_search_travel` 工具但 **尚未实现**，这是本阶段的核心战场。

### Task 3.1：文档切分与 Embedding 生成
- **学习目标**：理解 chunk 大小、overlap 对检索质量的影响；了解 embedding 模型选型。
- **项目实践**：
  - 将数据库中目的地数据（`destinations` 表 + `destination_features` 表）导出为文本片段。
  - 设计切分策略：按「目的地摘要」「美食条目」「美景条目」「文化条目」分别生成 chunk，保留元数据（destination_id、category）。
  - 接入 OpenAI Embedding API（或 MiniMax 兼容接口），为每个 chunk 生成向量。
- **关联文件**：新增 `src/rag/chunker.ts`、`src/rag/embedder.ts`
- **输出物**：切分后的 chunk 数据集；embedding 生成脚本及运行记录。

### Task 3.2：向量存储与检索
- **学习目标**：掌握向量数据库 CRUD 与相似度检索；理解 cosine similarity。
- **项目实践**：
  - 选择轻量方案：由于项目已用 MySQL，可先用**纯 MySQL 存储向量**（JSON 列存储 1536 维数组，用 SQL 计算内积近似相似度）作为 MVP；同时预留接口，后续可无缝切换到 Milvus/Chroma。
  - 新建向量表 `destination_embeddings(chunk_id, destination_id, category, text, embedding_json)`。
  - 实现 `searchSimilarChunks(query: string, topK: number)` 函数。
- **关联文件**：新增 `src/rag/vectorStore.ts`、数据库 migration 文件
- **输出物**：可运行的语义检索接口；Top-K 结果示例。

### Task 3.3：实现 `semantic_search_travel` 工具
- **学习目标**：理解「检索 → 拼 prompt → 生成」的 RAG 完整流程。
- **项目实践**：
  - 在 `src/agent/tools.ts` 中新增 `semantic_search_travel` 工具定义与实现。
  - 该工具内部调用 RAG 检索，返回相似 chunk 文本列表。
  - 更新系统提示词，确保模糊需求（如「想看雪山又不想太累」）优先走语义检索。
- **关联文件**：`src/agent/tools.ts`、`src/agent/prompts.ts`、`src/rag/` 下各模块
- **输出物**：Agent 可正确调用语义检索并基于检索结果回答。

### Task 3.4：混合检索与重排序
- **学习目标**：理解「关键词 + 向量混合检索」的优势；了解 rerank 的作用与代价。
- **项目实践**：
  - 实现混合检索：将 `search_destinations`（关键词 LIKE）与 `semantic_search_travel`（向量检索）的结果用 **RRF（Reciprocal Rank Fusion）** 融合。
  - 增加轻量重排序：用 cross-encoder 或简单的「查询词与结果文本的重合度」作为第二遍排序。
  - 在评测脚本中对比「纯关键词」「纯语义」「混合」三种策略的准确率。
- **关联文件**：`src/rag/hybridSearch.ts`、`scripts/eval-prompt.ts`
- **输出物**：混合检索流水线；三种策略的对比实验数据。

### Task 3.5：生成与溯源
- **学习目标**：减少幻觉的 prompt 技巧；让模型引用来源。
- **项目实践**：
  - 在系统提示词中增加约束：「回答时请标注信息来源（目的地名称及类型），如果不确定请说明」。
  - 在 API 响应中增加 `sources` 字段，返回引用的 chunk 列表。
- **关联文件**：`src/agent/prompts.ts`、`src/index.ts`
- **输出物**：带溯源信息的回答响应。

## 阶段4：Agent 核心（第5周 ~ 第6周，约2周）

### Task 4.1：增强 ReAct 循环与可观测性
- **学习目标**：深入理解 ReAct（推理 + 行动交错）的每一步；掌握 Agent 调试技巧。
- **项目实践**：
  - 在 `src/agent/llm.ts` 的 `runAgentWithTools` 中，增加**思考过程记录**：每轮将模型的 reasoning 文本（若有）存入独立字段。
  - 增加 `X-Agent-Trace-Id` 头部返回，将每轮的工具调用、参数、结果、耗时记录到日志，便于事后复盘。
- **关联文件**：`src/agent/llm.ts`
- **输出物**：可查看的 Agent 执行 trace；至少 3 个复杂查询的调试记录。

### Task 4.2：Plan-and-Execute 模式对比实现
- **学习目标**：掌握 Plan-and-Execute 与 ReAct 的差异；能在项目中切换模式。
- **项目实践**：
  - 新增 `src/agent/planner.ts`：实现「先规划再执行」模式——先用 LLM 生成任务计划（JSON 格式步骤列表），再按步骤执行。
  - 在 `src/index.ts` 中通过请求参数 `?mode=react|plan` 支持两种模式切换。
  - 对比实验：同一复杂问题（如「帮我规划一个7天云南游，包含美食和景点」）在两种模式下的步骤数、耗时、完成率。
- **关联文件**：新增 `src/agent/planner.ts`、`src/index.ts`
- **输出物**：可切换的双模式 Agent；对比实验报告。

### Task 4.3：记忆分层（长期记忆 + 用户画像）
- **学习目标**：掌握对话截断、摘要、用户画像的实现思路。
- **项目实践**：
  - **对话摘要**：当历史消息超过 `CHAT_HISTORY_LIMIT` 时，用 LLM 生成「用户偏好摘要」（如「用户喜欢自然风光、不喜欢太累的行程」），存储到 `chat_sessions` 表的 `summary` 字段。
  - **用户画像提取**：在 `src/agent/` 中增加 `memory.ts`，每次对话后更新用户偏好向量（关键词列表）。
  - 在构造 prompt 时，将摘要作为 system prompt 的一部分注入。
- **关联文件**：新增 `src/agent/memory.ts`、`src/db/chatRepo.ts`
- **输出物**：跨会话的记忆保持效果演示；摘要生成示例。

### Task 4.4：MCP（Model Context Protocol）入门接入
- **学习目标**：理解 MCP 与「普通 function calling」在架构上的区别。
- **项目实践**：
  - 将 `get_destination_detail` 工具封装为一个**最小 MCP Server**（使用 MCP SDK 或手写 stdio/sse 传输）。
  - Agent 侧实现 MCP Client，通过 MCP 协议调用该工具，替代本地直接调用。
  - 跑通官方或社区的一个外部 MCP Server 示例（如文件系统 MCP），让 Agent 具备读写本地文件的能力。
- **关联文件**：新增 `src/mcp/` 目录
- **输出物**：一个自研 MCP Server + 一个外部 MCP Server 接入；能口述两者架构差异。

## 阶段5：多智能体与工程化（第7周 ~ 第8周，约2周）

### Task 5.1：多 Agent 协作架构
- **学习目标**：理解 Supervisor 模式、角色分工、结构化消息传递。
- **项目实践**：
  - 新增 `src/agents/` 目录（注意复数，与单 agent 区分）：
    - `supervisor.ts`：意图识别 + 任务路由，不直接生成回答。
    - `ragAgent.ts`：专注于检索和答案生成（复用阶段3的 RAG 能力）。
    - `toolAgent.ts`：专注于外部工具调用（数据库查询、MCP 工具）。
  - Agent 间通过 JSON Schema 消息传递，Supervisor 负责整合结果。
  - 新增路由接口 `POST /orchestrate`，演示多 Agent 协作处理复杂问题。
- **关联文件**：新增 `src/agents/supervisor.ts`、`src/agents/ragAgent.ts`、`src/agents/toolAgent.ts`
- **输出物**：多 Agent 协作 Demo；单 Agent vs 多 Agent 的准确率/延迟对比。

### Task 5.2：模型路由与降级策略
- **学习目标**：掌握按任务类型、成本、延迟选择模型的策略。
- **项目实践**：
  - 在 `src/config.ts` 中配置多模型：`MODEL_LIGHT`（轻量模型，处理简单意图识别）、`MODEL_HEAVY`（强模型，处理复杂生成）。
  - 新增 `src/agent/router.ts`：根据查询复杂度（字数、关键词、历史模式）选择模型。
  - 实现降级：当强模型超时或报错时，自动切换到轻量模型 + 简化提示词。
- **关联文件**：新增 `src/agent/router.ts`、`src/config.ts`
- **输出物**：模型路由决策日志；降级触发记录。

### Task 5.3：容错、重试与可观测性
- **学习目标**：建立超时、重试、退避、幂等的工程意识。
- **项目实践**：
  - 在 `src/agent/llm.ts` 的 `postChat` 中增加：指数退避重试（3次）、超时控制（ configurable ）。
  - 工具调用失败时，不再直接返回 error JSON，而是尝试替代策略（如换一个检索方式）。
  - 增强日志：每个请求带 `trace_id`，记录总耗时、每步耗时、token 用量、模型名称、是否降级。
  - 新增 `GET /metrics` 接口，返回近 N 次请求的统计（平均延迟、错误率、token 消耗）。
- **关联文件**：`src/agent/llm.ts`、`src/index.ts`
- **输出物**：压力测试下的容错表现；metrics 接口返回的统计数据。

### Task 5.4：容器化与部署完善
- **学习目标**：掌握 Dockerfile、docker-compose、环境变量管理、简单 CI/CD。
- **项目实践**：
  - 优化现有 `docker-compose.yml`，增加应用服务容器（当前只有 MySQL）。
  - 编写 Dockerfile，支持多阶段构建（依赖安装 + 运行）。
  - 在 `.env.example` 中补充所有配置项注释；验证生产环境配置流程。
  - （可选）编写 GitHub Actions 工作流：提交时自动 build、lint。
- **关联文件**：`docker-compose.yml`、新增 `Dockerfile`、`.env.example`
- **输出物**：一键启动完整环境的 `docker compose up`；容器化部署笔记。

## 关键设计决策说明

1. **为什么从阶段2开始，阶段1只补 3~4 天？**
   项目已有 LLM 调用和 API 基础，阶段1的核心概念（Transformer、Token、参数）可通过「流式输出 + Token 统计」这两个小任务快速覆盖，不需要单独花整周时间。

2. **为什么 RAG 先用 MySQL 存向量而不是直接引入 Milvus？**
   学习路线建议「本地与云服务二选一练熟」。考虑到项目已有 MySQL 依赖，先用 MySQL 做 MVP 可以更快验证端到端流程；同时向量存储接口抽象好，后续切换到 Milvus/Chroma 只需换实现，不影响业务代码。这也对应面试问答集中「技术选型要兼顾迭代速度」的思路。

3. **为什么阶段4要同时保留 ReAct 和 Plan-and-Execute？**
   面试问答集 Q1 就是「为什么选择 ReAct 而不是 Plan-and-Execute」。项目中同时实现两种模式并支持切换，既能深入理解两者差异，又能在面试中用实际数据（步骤数、耗时、完成率）支撑回答。

4. **每个阶段的「输出物」是什么？**
   每阶段至少产出：① 可运行的代码 + ② 笔记/实验记录 + ③ 一个能对外演示的效果。阶段6 产出简历条目和面试故事。
